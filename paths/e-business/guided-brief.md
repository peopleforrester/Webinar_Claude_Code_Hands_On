# Guided Build: Meeting Prep Kit

Transform a messy meeting transcript into a set of stakeholder-ready artifacts — executive summary, decisions log, action items CSV, follow-up email draft, and a topic-by-topic timeline with sentiment. All generated by Claude. **No code.** The tool is the model; a Skill file encodes the format and voice rules; a shell script verifies the output.

**You do not write Python. You do not write JavaScript. You do not write any code at all.** You bring business judgment (what should a meeting prep kit contain, how should actions be attributed, what's the right tone for a follow-up email) — Claude does the extraction and writing.

**Time estimate:** 28 minutes

---

## Step 1: Set Up + Externalize (3 min)

Create your project directory and copy the sample transcript in:

```bash
mkdir meeting-prep-kit && cd meeting-prep-kit
cp <repo>/paths/e-business/sample-data/meeting-transcript.md .
```

Launch Claude and initialize:

```bash
claude
```

Inside the session:

```
/init
```

When Claude generates `CLAUDE.md`, press `#` at the start of a brand-new prompt and add a rule:

```
# This project processes meeting transcripts into a five-file prep kit in ./output/. The tool is Claude itself — do not write any Python, JavaScript, or other code to do extraction. Read the transcript and write the files directly.
```

Claude will ask which CLAUDE.md to save to — pick the project-level one.

Now copy the `meeting-prep-kit` Skill into your project:

```bash
!mkdir -p .claude/skills/meeting-prep-kit
!cp <repo>/skills-examples/meeting-prep-kit/SKILL.md .claude/skills/meeting-prep-kit/SKILL.md
```

> **Pillar 3 in action:** You just externalized your workflow into a Skill file. The format rules, voice rules, and done-criteria now live on disk and will be there the next time you launch Claude in this directory. Skills are scanned at session start — the Skill you just copied will be available after a restart, but for the guided build you'll reference its rules directly in your prompts.

## Step 2: Plan First (4 min)

Enter **Plan Mode** (`Shift+Tab` until `plan mode` shows in the input bar, or type `/plan`). Give Claude this prompt:

> Read `meeting-transcript.md` and the `meeting-prep-kit` Skill at `.claude/skills/meeting-prep-kit/SKILL.md`. Plan how you will process this specific transcript into the five required outputs: `summary.md`, `decisions.md`, `action-items.csv`, `followup-email.md`, `timeline.md`.
>
> In your plan, list:
> - Every decision you identified in the transcript (with rationale or "Not stated")
> - Every action item you identified (with owner and due date or TBD)
> - Every open question
> - Every parked item
> - Every topic the timeline will cover, in order
> - Any position shifts or sentiment notes worth flagging
>
> Do not write any files yet. This is planning only.

Review Claude's plan. Things to check:
- Did it catch the $12K incentive budget ask?
- Did it pick up Dana joining late and Sarah's recap?
- Did it flag the parked items (load test env, launch comms owner)?
- Did it identify the Alex-Priya v1/v2 disagreement as a position shift or tense moment?
- Does every action item have an owner and a date?

If the plan misses things, tell Claude what's missing and have it update the plan before moving on. **This is where the business judgment lives** — you're shaping what "good" looks like before any output is written.

> **Pillar 2 in action:** You planned before generating. The plan is the contract — if it's wrong, you catch it for free, before any file exists. This is the same rule-plus-check discipline the engineering paths use, just applied to document work.

---

## Step 3: Build + Verify (12 min)

Exit Plan Mode (`Shift+Tab` until `plan mode` disappears). Before running the Skill, create the verification script — this is the deterministic check that catches format violations:

> Create a shell script `verify.sh` in the project root that checks the five output files. It must verify:
>
> 1. All five files exist: `output/summary.md`, `output/decisions.md`, `output/action-items.csv`, `output/followup-email.md`, `output/timeline.md`
> 2. `output/action-items.csv` first line is exactly `owner,action,due_date,status,source_quote`
> 3. Every data row in `action-items.csv` has a non-empty first field (owner) and a non-empty third field (due_date)
> 4. `output/summary.md` is under 300 words
> 5. None of the five output files contain any of these weasel words (case-insensitive): `may`, `might`, `could`, `perhaps`, `possibly`, `we discussed`, `it was mentioned`, `moving forward`
>
> Print `OK` if all checks pass. For any failure, print `FAIL: <specific check>` on its own line and exit non-zero. Make the script executable.

Sanity check it exists and is executable:

```
!ls -la verify.sh
```

Now add a rule that makes verify.sh run automatically after every output change. Type `#` at the start of a new prompt:

```
# After generating or updating any file in ./output/, always run ./verify.sh and confirm it passes before responding.
```

Save to the project-level CLAUDE.md.

Now run the Skill:

> Using the `meeting-prep-kit` skill, process `meeting-transcript.md` into the five required output files in `./output/`. Follow every rule in the Skill file. When you are done, run `./verify.sh` and report the result.

Watch for:
- Claude creates `output/` if it doesn't exist
- Claude writes all five files
- Claude runs `./verify.sh` automatically because of the CLAUDE.md rule
- If `verify.sh` fails, Claude sees the failure and self-corrects (common failures: weasel word slipped in, CSV row missing a date, summary slightly over 300 words)

Once `verify.sh` prints `OK`, open the generated files yourself and read them. Ask:
- Would I actually send `followup-email.md` to attendees with minor edits, or does it need a rewrite?
- Does `timeline.md` catch the *feel* of the meeting, or is it bloodless?
- Is `action-items.csv` clean enough to paste into a tracker?

If any of these are "no," tell Claude specifically what to fix. This is the iteration loop.

> **Pillar 4 in action:** The rule-plus-check pattern. A rule in CLAUDE.md (run verify after every output change) paired with a deterministic script (`verify.sh`) means format violations get caught and fixed automatically. Claude can *see* the failure and act on it — that's the whole reliability story.

## Step 4: Iterate + Verify (5 min)

Now use the tool the way a PM actually would. Pick two or three of these:

- **Executive version:** "Generate an `output/summary-exec.md` that is under 100 words — the three things a VP needs to know, nothing else. Run verify.sh after."
- **Slack version:** "Generate an `output/slack-update.md` — a Slack-formatted update for the #portal-team channel. Bullet points only, no headers, under 200 words, active voice. Run verify.sh after."
- **Commitments to one person:** "Generate `output/commitments-alex.md` — just the action items where Alex is the owner, formatted as a personal to-do list with context. Run verify.sh after."
- **Risk radar:** "Generate `output/risks.md` — every risk or concern raised in the transcript, with source quote, even if it was parked or waved off. Separate them into 'flagged as risk' and 'waved off, re-examine later'. Run verify.sh after."

Each of these is a new output file, but the same transcript and the same Skill. You're getting five different deliverables from one piece of judgment work.

> **Pillar 4 in action (again):** Every new deliverable goes through the same verification. The CLAUDE.md rule makes it automatic — you don't re-remind Claude.

## Step 5: Create Your Own Skill (2 min)

The `meeting-prep-kit` skill is generic. Your org has its own meeting style — maybe you use RACI, maybe your action items include a Jira link, maybe your follow-up emails always open with "Hi team." Create a second skill that captures *your* conventions:

> Create a skill file at `.claude/skills/my-meeting-style/SKILL.md`. It must start with YAML front matter between `---` fences, with two fields: `name: my-meeting-style` and `description: {one sentence describing what this skill enforces}`. After the closing `---`, add instructions covering: (1) the exact phrasing and section headers I use in my follow-up emails, (2) whether action items should link to a tracker, (3) my preferred tone (formal vs casual), and (4) any org-specific vocabulary or stakeholder titles to preserve verbatim.

Claude will ask you interactive questions if it needs detail. This Skill is yours — it will ride with you into every future project that copies it.

> **Pillar 3 in action:** You externalized *your* conventions. Every future meeting reuses the same rules without re-explaining them.
